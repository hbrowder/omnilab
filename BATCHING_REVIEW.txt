PROBLEM
────────────────────────────────────────────────────────────────────────────
High-volume packet captures (100+ packets/sec) were flooding the WebSocket
and frontend UI with individual messages, causing:
- 100+ WebSocket messages per second
- UI animation queue overload
- Poor performance during live captures
- Browser tab slowdown/freezing

SOLUTION
────────────────────────────────────────────────────────────────────────────
Implemented event batching that queues packets in memory and flushes them
every 100ms, reducing message rate from 100+ msgs/sec to ~10 msgs/sec.

────────────────────────────────────────────────────────────────────────────

HOW IT WORKS
────────────────────────────────────────────────────────────────────────────
1. Packet arrives from tcpdump
2. Create event dict (filter_id, link_id, packet_summary)
3. Append to pending_events queue (thread-safe with lock)
4. Check if 100ms elapsed since last flush
5. If yes: flush up to 20 events as ONE WebSocket message
6. Reset timer and continue

Key features:
- Thread-safe: Lock prevents race conditions across multiple interfaces
- Graceful shutdown: Flushes remaining events when capture stops
- Burst protection: Max 20 events per batch prevents oversized messages
- Smooth output: Converts bursty input to steady 10 msgs/sec output

NEW MESSAGE TYPE
────────────────────────────────────────────────────────────────────────────
{
  "type": "traffic_batch",
  "events": [
    {"filter_id": "...", "link_id": "...", "packet_summary": "..."},
    {"filter_id": "...", "link_id": "...", "packet_summary": "..."},
    ... (1-20 events)

  "count": 6,

Old "traffic_match" format still works (backward compatible).

CONFIGURATION (FUTURE)
────────────────────────────────────────────────────────────────────────────
Current settings work well for typical network lab traffic:
- batch_interval: 100ms (10 batches/sec)
- max_per_batch: 20 events

Could be made configurable later for different use cases:
- High-speed testing: 50ms interval (20 batches/sec, more responsive)
- Low-bandwidth: 200ms interval (5 batches/sec, fewer messages)
- Large labs: 50 events per batch (handles burst traffic better)
